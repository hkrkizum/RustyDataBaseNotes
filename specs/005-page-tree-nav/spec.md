# Feature Specification: Page Tree Navigation

**Feature Branch**: `005-page-tree-nav`
**Created**: 2026-03-22
**Status**: Draft
**Input**: User description: "ページ階層（親子関係）+ サイドバーナビゲーション + Tailwind CSS / shadcn/ui 全面移行"

## User Scenarios & Testing

### User Story 1 - Unified Visual Design (Priority: P1)

ユーザーはアプリケーション全体（エディタ，テーブルビュー，および後続ストーリーで追加される
サイドバー）で統一されたビジュアルデザインを体験する。ボタン，入力フィールド，パネル，
タイポグラフィが共通のデザインシステムに従い，OS のカラーテーマ（ライト/ダーク）に適応する。
既存の全画面がこのデザインシステムに移行される。

**Why this priority**: 統一されたデザインシステムは後続のサイドバー・ツリー UI の構築基盤である。先に移行することで，新規コンポーネントと既存コンポーネントのスタイル二重管理を回避し，一貫した UX を最初から保証する。

**Independent Test**: エディタとテーブルビューの各画面でボタン・入力・スペーシング・色が一貫していることをビジュアルレビューで確認。ダークモードでも同様。

**Acceptance Scenarios**:

1. **Given** アプリの任意の画面, **When** ボタン・入力・パネルを確認, **Then** 共通のカラーパレット・スペーシング・角丸が適用されている
2. **Given** ライトモードの OS, **When** アプリを開く, **Then** 全画面がライトテーマで描画される
3. **Given** ダークモードの OS, **When** アプリを開く, **Then** 全画面がダークテーマで描画される
4. **Given** 既存のページ一覧・エディタ・テーブルビュー, **When** デザインシステム移行後に操作する, **Then** すべての既存機能が以前と同じように動作する（機能の退行なし）
5. **Given** デザインシステム移行が完了した状態, **When** プロジェクト内のスタイルファイルを確認, **Then** 旧スタイリング方式のファイル（CSS Modules 等）がゼロであり，デザインシステムのみでスタイルが管理されている

---

### User Story 2 - Sidebar Navigation (Priority: P2)

ユーザーはアプリケーションを開くと，左側に常時表示されるサイドバーを見る。
サイドバーにはすべてのスタンドアロンページとデータベースが表示される。
どの画面（エディタ，テーブルビュー）にいても，サイドバーのアイテムをクリックするだけで
直接遷移でき，ホーム画面に戻る必要がなくなる。

**Why this priority**: 現在のナビゲーションはリストビューへの「戻る」操作が必須であり，コンテンツが増えるほど操作効率が低下する。サイドバーはすべての将来機能（検索，複数ビュー等）の UX 基盤となる。US1 のデザインシステム上に構築される。

**Independent Test**: ユーザーが5つのページと2つのデータベースを作成し，サイドバーからクリックだけですべてのアイテム間を遷移できることを確認する。

**Acceptance Scenarios**:

1. **Given** アプリが起動している, **When** 画面左側を見る, **Then** すべてのスタンドアロンページとデータベースがサイドバーに表示されている
2. **Given** ページを編集中, **When** サイドバーで別のページをクリック, **Then** そのページのエディタに直接遷移する
3. **Given** ページを編集中, **When** サイドバーでデータベースをクリック, **Then** そのデータベースのテーブルビューに直接遷移する
4. **Given** テーブルビューを表示中, **When** サイドバーでページをクリック, **Then** そのページのエディタに直接遷移する

---

### User Story 3 - Page Hierarchy (Priority: P3)

ユーザーはスタンドアロンページを他のページの下にネストし，ツリー構造で整理できる。
サイドバーでは親ページの配下にインデント表示され，展開/折りたたみで表示を管理できる。
ページのドラッグ＆ドロップで親子関係を変更できる。

**Why this priority**: ページのフラットなリストは蓄積するほど管理不能になる。ツリー構造はユーザーが自然に情報を分類する方法を反映する。US2 のサイドバー上にツリー機能を追加する。

**Independent Test**: ユーザーが親ページを作成し，その下に3つのサブページを作成。親を折りたたんでサブページが非表示になり，展開して再表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** スタンドアロンページがある, **When** そのページの下にサブページを作成, **Then** サブページがサイドバーで親の下にインデント表示される
2. **Given** 子ページを持つ親ページがある, **When** 展開/折りたたみトグルをクリック, **Then** 子ページの表示/非表示が切り替わる
3. **Given** 任意の階層にあるページ, **When** ドラッグ＆ドロップで別の親の下に移動, **Then** サイドバーが新しい位置を反映して更新される
4. **Given** 子ページを持つ親ページ, **When** 親ページを削除, **Then** 子ページは親の親（またはルートレベル）に昇格し，データは一切失われない
5. **Given** ページが最大ネスト深度にある, **When** その下にサブページを作成しようとする, **Then** 深度制限のエラーメッセージが表示され操作は拒否される
6. **Given** ページ A が B の子, **When** B を A の子に移動しようとする, **Then** 循環参照のエラーメッセージが表示され操作は拒否される

---

### Edge Cases

- **アプリクラッシュ時**: ページ階層の変更（移動，親削除）はアトミックに実行される。クラッシュ時は変更前の状態が保持される。
- **データベースページの階層参加**: データベースに所属するページ（`database_id` を持つ）はページ階層に参加できない。子ページを持つことも，他のページの子になることもできない。サイドバーではデータベースエントリの配下に表示される。
- **大量ページ**: 数百ページのツリーでもサイドバーの応答性が維持される。
- **循環参照**: ドメイン層で循環参照を検出し，操作前に拒否する。
- **サイドバーの展開状態**: ツリーの展開/折りたたみ状態は localStorage でセッション間保持され，アプリ再起動後も同じ状態が復元される。
- **サイドバーの空状態**: ページもデータベースも存在しない場合，サイドバーはヘッダーと新規作成ボタンのみを表示する（特別な空状態 UI は設けない）。

## Requirements

### Functional Requirements

- **FR-001**: アプリケーションの左側に常時表示のサイドバーを表示しなければならない（MUST）。サイドバーはエディタ画面・テーブルビュー画面を含むすべての画面で表示される。
- **FR-002**: サイドバーでページをクリックするとそのページのエディタに，データベースをクリックするとそのテーブルビューに直接遷移しなければならない（MUST）。現在アクティブなアイテムは背景色ハイライト（shadcn/ui 標準スタイル）で視覚的に示す。
- **FR-003**: スタンドアロンページ同士の親子関係をサポートしなければならない（MUST）。ページを他のページの子として作成できる。
- **FR-004**: ページのネスト深度を最大5階層に制限しなければならない（MUST）。
- **FR-005**: 循環する親子参照を防止しなければならない（MUST）。
- **FR-006**: サイドバーはネストされたページを折りたたみ可能なツリーとして表示しなければならない（MUST）。
- **FR-007**: ドラッグ＆ドロップによるページの移動（親変更・ルートへの昇格）をサポートしなければならない（MUST）。ドラッグ中は有効なドロップ先をハイライト背景＋挿入位置インジケーターラインで示し，無効なドロップ先（循環参照・深度超過・データベースページ）は禁止カーソルで示す。
- **FR-008**: 親ページ削除時，子ページは削除された親の親（またはルート）に昇格しなければならない（MUST）。子ページのデータは一切削除されない。
- **FR-009**: データベースに所属するページはページ階層に参加してはならない（MUST NOT）。子ページを持てず，他のページの子にもなれない。
- **FR-010**: サイドバーの表示/非表示を切り替えられなければならない（MUST）。トグル操作はサイドバー上部のトグルボタンおよびキーボードショートカット（Cmd/Ctrl+B）の両方で行える。表示/非表示の状態は localStorage でセッション間保持する。サイドバーの幅は固定（240–260px）とし，ドラッグによるリサイズは本スコープに含まない。
- **FR-011**: データベースとその所属ページはサイドバーでグループ化して表示しなければならない（MUST）。データベースエントリの下にそのデータベースのページが表示される。データベースグループは展開/折りたたみ可能とし，その状態はページツリーと同じ仕組みで localStorage に永続化する。ページとデータベースはアイコンで視覚的に区別する（ページ=ドキュメントアイコン，データベース=テーブルアイコン，Lucide Icons を使用）。ルートレベルではスタンドアロンページとデータベースをセクション分離せず created_at 順で混在表示する。
- **FR-012**: ローカルファースト動作を維持し，外部サービスを必要としてはならない（MUST NOT）。
- **FR-013**: サイドバーの新規作成ボタンからページとデータベースの両方を作成できなければならない（MUST）。種別選択 UI（ドロップダウン等）を提供する。ページ作成はデフォルトでルートレベルのスタンドアロンページとする。新規作成後は作成されたアイテムに自動遷移し，サイドバー上でインライン名前編集を即時開始する。親ページのコンテキストメニューからの作成はその親の子として作成する。コンテキストメニューは「子ページ作成」「名前変更」「削除」の3アクションを提供する。コンテキストメニューはアイテムホバー時に表示される「...」ボタンのクリック，または右クリックで開く。「名前変更」はサイドバー内インライン編集（ラベルがテキスト入力に変化し，Enter で確定，Esc でキャンセル）で行う。移動操作はドラッグ＆ドロップ（FR-007）でカバーするためコンテキストメニューには含まない。
- **FR-014**: ツリーの展開/折りたたみ状態をセッション間で保持しなければならない（MUST）。初回起動時（localStorage 未設定）のデフォルトは全ノード折りたたみ（ルートレベルのみ表示）とする。
- **FR-015**: Tailwind CSS + shadcn/ui をデザインシステムとして採用し，アプリケーション全体で統一されたビジュアルデザイン（カラーパレット，タイポグラフィ，スペーシング，ボーダースタイル）を適用しなければならない（MUST）。移行完了後，旧スタイリング方式のファイル（CSS Modules 等）をプロジェクトに残してはならない（MUST NOT）。
- **FR-016**: ライトモードとダークモードの両方をサポートし，OS のシステム設定に追従しなければならない（MUST）。
- **FR-017**: アプリ起動時に最後に開いたページまたはデータベースを自動的に復元しなければならない（MUST）。最後に開いたアイテムの ID は localStorage で永続化する。該当アイテムが削除済みの場合はルートレベルの先頭アイテムにフォールバックする。アイテムが一つも存在しない場合は空状態を表示する。

### Key Entities

- **Page（拡張）**: 既存の属性（id, title, database_id, created_at, updated_at）に加え，`parent_id`（任意，自己参照）および `sort_order`（INTEGER DEFAULT 0）を持つ。`sort_order` は将来の手動並べ替え機能に備えて今回のマイグレーションで追加するが，本スコープでは表示順に `created_at DESC` を使用する。ページは「スタンドアロン（database_id なし，parent_id あり/なし）」または「データベース所属（database_id あり，parent_id なし，子なし）」のいずれか。
- **Sidebar State（新規）**: ツリーの展開/折りたたみ状態およびサイドバーの表示/非表示状態を表す。localStorage に保持されるフロントエンド側のローカル設定（DB スキーマ変更なし）。

## Constraints & Compliance

- **CC-001 Data Integrity**: ページ階層の変更（子作成・移動・親削除）はアトミックに実行される。クラッシュ時に以前の状態が復旧可能。`parent_id` は外部キー制約で参照整合性を保証する。
- **CC-002 Privacy**: 本機能はいかなる外部通信・テレメトリ・外部依存を導入しない。すべてのデータはローカルに保持される。
- **CC-003 Performance**: サイドバーは500ページのツリーで200ms以内にレンダリングされる。ツリーの展開/折りたたみは50ms以内。サイドバークリックからの画面遷移は100ms以内。
- **CC-004 Boundary Types**: `parent_id` はバリデーション済みの PageId として型付き IPC コマンドで公開される。フロントエンドは `parentId: string | null` を含む DTO を受信する。エラー条件（循環参照，深度超過，データベースページのネスト）は型付きエラーバリアントとして返される。
- **CC-005 Testability**: ドメイン層テストで検証: 循環参照検出，深度制限，親削除時の子昇格，データベースページの階層不参加。統合テストで検証: サイドバーのツリー描画，任意画面からのナビゲーション。ビジュアルレビューでデザイン一貫性を検証。

## Assumptions

- 最大ネスト深度5は個人のノート整理に十分である
- サイドバーの展開/折りたたみ状態は localStorage で保持する（SQLite やファイルは使用しない）
- ダークモード切り替えは OS のシステム設定に従い，アプリ内での手動切り替えは本スコープに含まない
- サイドバー内のページ表示順は作成日時降順を基本とし，ドラッグによる手動並び替えは本スコープでは階層変更（親子関係の変更）に限定する。同一階層内の表示順の手動変更は後続スコープとする
- データベースに所属するページは構造化データの「行」であり，ノートの階層構造とは概念が異なるため，ページ階層には参加しない
- アプリ起動時は最後に開いたページ/データベースを自動復元する。該当アイテムが存在しない場合はルートレベル先頭アイテムにフォールバックする
- エディタは本スコープで自動保存（debounce 付き）に移行する。既存の手動保存（Ctrl+S / 保存ボタン）＋ UnsavedConfirmModal を廃止し，プロパティと同様の自動保存方式に統一する。画面遷移時の未保存確認ダイアログは不要になる。自動保存失敗時はサイレントリトライ（最大3回）し，全リトライ失敗時に toast で警告する
- サイドバーツリーのキーボードナビゲーション（矢印キー移動，Enter 選択，WAI-ARIA TreeView パターン）は本スコープに含まない。後続スコープで検討する

## Success Criteria

### Measurable Outcomes

- **SC-001**: ユーザーがサイドバーから1クリックで任意のページ/データベースに遷移でき，ホーム画面への「戻る」操作が不要になる
- **SC-002**: ユーザーが50以上のページを最大5階層のツリー構造で整理できる
- **SC-003**: 500ページのツリーでサイドバーが200ms以内にレンダリングされる
- **SC-004**: 親ページ削除時に子ページが一切失われない（データ損失ゼロ）
- **SC-005**: サイドバー・エディタ・テーブルビューのすべてのビジュアルレビューで色・タイポグラフィ・スペーシングの一貫性が確認される
- **SC-006**: ライト/ダークの両カラースキームで全画面が正しく描画される
- **SC-007**: 品質ゲート（フォーマット，lint，テスト，ドキュメントビルド）をすべて通過する
