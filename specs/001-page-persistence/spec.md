## Clarifications

### Session 2026-03-21

- Q: フロントエンドフレームワークは React と Vue のどちらを採用するか？ → A: React（TSX）
- Q: Page の識別子に使用する UUID のバージョンは？ → A: UUIDv7（時刻順ソート可能）
- Q: Page のタイトルに最大文字数の制約を設けるか？ → A: 最大 255 文字
- Q: エラー発生時のフロントエンド表示方式は？ → A: Toast 通知（自動消去あり，非ブロッキング）

### User Story 1 - ページを新規作成する (Priority: P1)

ユーザーはアプリケーションを起動し，新しいページを作成できる。
作成されたページにはタイトルが付与され，次回以降の起動でも失われない。
これはアプリケーションの最も基本的な操作であり，すべての後続機能の土台となる。

**Why this priority**: ページの作成と永続化はノートブックアプリの存在意義そのものであり，
これが動作しなければ他のすべての機能（ブロック編集，データベース，ビュー）が成立しない。

**Independent Test**: タイトルを指定してページを作成し，アプリケーションを再起動した後も
そのページが一覧に表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** アプリケーションが起動している状態，**When** ユーザーがタイトルを入力してページ作成を実行する，**Then** 新しいページが作成され一覧に表示される
2. **Given** ページが作成された状態，**When** アプリケーションを終了し再起動する，**Then** 作成済みのページが一覧に表示される
3. **Given** アプリケーションが起動している状態，**When** ユーザーがタイトルを空のまま作成を試みる，**Then** エラーメッセージが表示され，ページは作成されない

---

### User Story 2 - ページ一覧を閲覧する (Priority: P1)

ユーザーはアプリケーションを開いたとき，保存済みのすべてのページを一覧として確認できる。
一覧にはページのタイトルと作成日時が表示される。

**Why this priority**: ページの作成と一覧表示は相互に依存する最小の機能セットであり，
一方だけでは実用にならない。

**Independent Test**: 複数のページを作成した後，一覧画面でそれらすべてが
タイトルと作成日時付きで表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数のページが保存されている状態，**When** ユーザーがアプリケーションを開く，**Then** すべてのページがタイトルと作成日時とともに一覧表示される
2. **Given** ページがまだ一つも存在しない状態，**When** ユーザーがアプリケーションを開く，**Then** 空の状態であることが分かる表示がされ，ページ作成を促すガイダンスが示される

---

### User Story 3 - ページのタイトルを更新する (Priority: P2)

ユーザーは既存のページのタイトルを変更できる。変更は永続化され，
次回以降の起動でも反映される。

**Why this priority**: タイトル更新は CRUD の基本操作であり，後続のブロック編集機能への
足がかりとなるが，作成・一覧より優先度は低い。

**Independent Test**: 既存ページのタイトルを変更し，アプリケーション再起動後に
変更後のタイトルが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 既存のページがある状態，**When** ユーザーが一覧画面上でタイトルをクリックしインライン編集モードに入り，タイトルを変更して確定する，**Then** 一覧に変更後のタイトルが反映される
2. **Given** 既存のページがある状態，**When** ユーザーがインライン編集でタイトルを空に変更しようとする，**Then** エラーが表示され，元のタイトルが維持される

---

### User Story 4 - ページを削除する (Priority: P2)

ユーザーは不要になったページを削除できる。削除されたページは一覧から消え，
永続化層からも除去される。

**Why this priority**: 削除は CRUD を完成させる操作であり，データの整理に必要だが，
作成・一覧より優先度は低い。

**Independent Test**: ページを削除した後，一覧から消えていること，
かつアプリケーション再起動後も復活しないことを確認する。

**Acceptance Scenarios**:

1. **Given** 既存のページがある状態，**When** ユーザーがそのページの削除を実行する，**Then** ページが一覧から消える
2. **Given** ページを削除した状態，**When** アプリケーションを再起動する，**Then** 削除したページは一覧に表示されない
3. **Given** 削除操作を開始した状態，**When** ユーザーが削除ボタンを押す，**Then** 確認ダイアログ（モーダル）が表示され，ユーザーが承認した場合のみ削除が実行される

### Edge Cases

- アプリケーションがページの保存途中にクラッシュした場合，データが中途半端な状態にならず，直前の整合状態が維持される
- 永続化層のファイルが外部から削除・破損された場合，アプリケーションは起動時にエラーを報告し，可能な限りの復旧手段を提示する
- ページ数が数千件に達しても，一覧表示の応答性が実用的な範囲に留まる
- マイグレーションが途中で失敗した場合，データベースが破損せず，再試行可能な状態が維持される

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは，タイトルを持つページを新規作成し，一意な識別子（UUIDv7）と作成日時・更新日時を自動付与して永続化できなければならない（MUST）
- **FR-002**: システムは，保存済みのすべてのページをタイトル・作成日時とともに一覧取得できなければならない（MUST）
- **FR-003**: システムは，識別子を指定して個別のページ情報を取得できなければならない（MUST）
- **FR-004**: システムは，既存ページのタイトルを更新し，更新日時を自動的に記録できなければならない（MUST）
- **FR-005**: システムは，指定されたページを永続化層から完全に削除できなければならない（MUST）
- **FR-006**: システムはローカル完結で動作し，ページの作成・取得・更新・削除のいずれにおいても外部サービスへの通信を行ってはならない（MUST NOT）
- **FR-007**: すべてのデータ操作は明確なエラー情報を返し，操作の成否をユーザーが判別できなければならない（MUST）。フロントエンドでは Toast 通知（自動消去あり，非ブロッキング）によりエラーを表示する
- **FR-008**: 永続化層はバージョン管理されたマイグレーションにより構造変更が可能でなければならない（MUST）
- **FR-009**: タイトルが空または 255 文字を超えるページは作成・更新できてはならない（MUST NOT）
- **FR-010**: フロントエンド（React）は，ページの一覧表示，新規作成，タイトル更新，削除のすべての CRUD 操作の UI を提供しなければならない（MUST）

### Key Entities

- **Page**: ノートブックの基本単位。一意な識別子（UUIDv7，時刻順ソート可能），タイトル（空文字列不可，最大 255 文字），作成日時，更新日時を持つ。ライフサイクルは「作成 → 更新（0回以上） → 削除」であり，作成後は識別子が不変。将来的にはブロックの集合を内包するコンテナとなるが，本スライスではメタデータのみを扱う

## Constraints & Compliance *(mandatory)*

- **CC-001 Data Integrity**: ページの書き込み（作成・更新・削除）は単一のトランザクション内で完結し，途中失敗時にはロールバックにより直前の整合状態を維持する。マイグレーションもトランザクション保護下で実行される
- **CC-002 Privacy**: 本機能はすべてローカルファイルシステム内で完結し，ネットワーク通信，テレメトリ送信，外部サービス呼び出しを一切行わない
- **CC-003 Performance**: ページ数 1,000 件以下の環境において，一覧取得がユーザーに体感遅延を与えない応答速度を維持する。アプリケーション起動時のマイグレーション処理もユーザーの待機感を生じさせない範囲に収める
- **CC-004 Boundary Types**: バックエンド（Rust）とフロントエンド（React + TypeScript）の間は型付き IPC 契約で接続し，ページの構造と操作結果を両側で型安全に扱う。エラーはフロントエンドが表示可能な構造化された形式で伝達する
- **CC-005 Testability**: バックエンド側のデータ操作ロジックは，インメモリデータベースを用いた自動テストで検証可能とする。各ユーザーストーリーの受入シナリオが自動テストとして表現可能であること

## Assumptions

- 本スライスではページのメタデータ（タイトル，日時）のみを扱い，ページ内のブロックやコンテンツ本体は後続の機能で実装する
- ユーザーは単一であり，認証・認可の仕組みは不要
- ページの並び順は作成日時の降順（新しい順）をデフォルトとする
- 削除は物理削除（論理削除やゴミ箱機能は後続の機能で検討）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがタイトルを入力してからページが一覧に表示されるまで，体感的に即座（1 秒以内）に完了する
- **SC-002**: 1,000 件のページが保存された状態で，一覧表示が 1 秒以内に完了する
- **SC-003**: 保存中のクラッシュや異常終了の後，アプリケーション再起動時にデータが破損していない状態が維持される
- **SC-004**: すべてのデータ操作（作成・取得・更新・削除）の品質ゲート（整形，lint，自動テスト，ドキュメントビルド）が通過する
