## Constitution 準拠チェック

### カバーされている原則

| Constitution 原則 | 対応するチェック項目 |
|------------------|-------------------|
| I. Local-First Product Integrity | DI-CHK007（トランザクション原子性），DI-CHK011（マイグレーションロールバック），DI-CHK015（操作単位の原子性），DI-CHK028（データ破損リカバリ） |
| II. Domain-Faithful Information Model | 3 チェックリスト全体で view, database, page, property の語彙を一貫使用。チェック項目自体がドメインモデルの整合性を検証 |
| III. Typed Boundaries and DDD | TO-CHK015-018（型別演算子バリデーション），DI-CHK018（エラー型の識別可能性），DI-CHK021（契約と仕様の整合） |
| IV. Test-First Delivery | DI-CHK004（受入シナリオの網羅性），UX-CHK030-032（受入シナリオの操作フロー網羅性） |
| V. Safe Rust / SOLID / Maintainability | TO-CHK019（Assumptions vs FR の配置適正性 — YAGNI/保守性に関連） |
| VII. Defensive Error Handling | DI-CHK017（書き込み失敗時の通知），DI-CHK028（JSON デシリアライズ失敗），TO-CHK016（エラーメッセージのコンテキスト） |

### カバーされていない原則

以下の Constitution 原則に対応するチェック項目が不足しています:

- **VI. Rust ドキュメント標準**: 3 チェックリストのいずれにも公開 API ドキュメントの品質を検証する項目がない。ただし本チェックリストのスコープ（データ整合性・型別ロジック・UX）では直接的な関連が薄く，別途 `doc-quality.md` チェックリストで対応すべき領域
- **IV. Test-First（テスト戦略の詳細）**: 受入シナリオの存在はチェックしているが，Red-Green-Refactor のワークフロー遵守や，プロパティ型ごとの独立テスト可能性（CC-005）の仕様品質は未検証
