# Feature Specification: ブロックエディタ

**Feature Branch**: `002-block-editor`
**Created**: 2026-03-21
**Status**: Draft
**Input**: ページの中身を編集できるようにする。ページを開いてプレーンテキストブロックを追加・編集・並び替え・削除し，明示的に保存できるようにする。

## Clarifications

### Session 2026-03-21

- Q: 初期スキーマに `block_type` カラムを含めるか？ → A: `block_type TEXT NOT NULL DEFAULT 'text'` を初期スキーマに含める
- Q: ブロックの UUIDv7 はフロントエンド／バックエンドのどちらで生成するか？ → A: バックエンド（Rust）で生成。ブロック追加時点で即座に UUIDv7 を割り当て，フロントエンドには常に正式 ID を返す（仮 ID は不要）
- Q: ブロック操作のロジックはフロントエンドとバックエンドのどちらに置くか？ → A: バックエンド（Rust）に EditorSession としてインメモリ状態を持たせ，全操作（追加・編集・並び替え・削除）を Rust のドメインロジックで処理する。フロントエンドは薄い UI 層として IPC コマンドを呼び出し，返却された EditorState で画面を更新する。理由: Rust の型安全性とテスト容易性を最大活用し，ビジネスロジックの二重実装を避けるため

### User Story 1 - ページを開いてブロック一覧を表示する (Priority: P1)

ユーザーがページ一覧画面でページタイトルをクリックすると，エディタ画面に切り替わり，
そのページに属するブロックが順序どおりに表示される。エディタ画面にはページ一覧に
戻るためのボタンが表示される。ブロックがまだ存在しないページでは，空のエディタ画面と
ブロック追加の案内が表示される。

**Why this priority**: ブロックの編集・操作はすべてエディタ画面上で行われるため，
画面遷移がなければ他のすべてのユーザーストーリーが成立しない。

**Independent Test**: ブロックを持つページと持たないページの両方を開き，
画面切り替えと一覧への復帰が正しく動作することを確認する。

**Acceptance Scenarios**:

1. **Given** ブロックを持つページが存在する, **When** ユーザーがページ一覧でそのタイトルをクリックする, **Then** エディタ画面に切り替わり，ブロックが position 順に表示される
2. **Given** ブロックを持たないページが存在する, **When** ユーザーがそのページを開く, **Then** 空状態の案内メッセージとブロック追加手段が表示される
3. **Given** エディタ画面が表示されている, **When** ユーザーが戻るボタンをクリックする, **Then** ページ一覧画面に戻り，ページ一覧が最新状態で表示される

---

### User Story 2 - テキストブロックを追加する (Priority: P1)

ユーザーがエディタ画面でブロック追加操作を行うと，新しいプレーンテキストブロックが
一覧の末尾に追加される。追加されたブロックは即座に編集可能な状態になる。
この時点ではまだ永続化されておらず，未保存状態として表示される。

**Why this priority**: ブロックを追加できなければページに内容を持たせられないため，
エディタの基本機能として不可欠。

**Independent Test**: エディタ画面でブロックを追加し，ブロックが末尾に表示され
編集可能になることを確認する。未保存インジケータが表示されることも確認する。

**Acceptance Scenarios**:

1. **Given** エディタ画面が表示されている, **When** ユーザーがブロック追加操作を行う, **Then** 新しい空のテキストブロックが一覧の末尾に追加され，フォーカスが当たる
2. **Given** ブロックが 0 件のページ, **When** 最初のブロックを追加する, **Then** 空状態の案内が消え，ブロックが表示される
3. **Given** ブロックを追加した直後, **When** 画面の状態を確認する, **Then** 未保存インジケータが表示されている

---

### User Story 3 - ブロックの内容を編集する (Priority: P1)

ユーザーがブロックのテキスト領域に入力すると，その内容がリアルタイムに画面上で
反映される。編集はローカル状態のみを変更し，永続化は行わない。
内容が変更されると未保存インジケータが表示される。

**Why this priority**: ブロックの内容を変更できることがエディタとしての根幹機能であり，
追加と同等の重要度を持つ。

**Independent Test**: 既存ブロックの内容を変更し，画面上で反映されること，
未保存インジケータが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** テキストブロックが表示されている, **When** ユーザーがテキストを入力する, **Then** ブロックの内容が画面上で即座に更新される
2. **Given** 保存済みの状態, **When** ブロック内容を変更する, **Then** 未保存インジケータが表示される
3. **Given** ブロックの内容が 10,000 文字に達している, **When** さらに入力を試みる, **Then** 入力が制限され，上限に達した旨が通知される

---

### User Story 4 - ブロックを並び替える (Priority: P2)

ユーザーがブロックの上移動・下移動ボタンを操作すると，ブロックの表示順序が変更される。
順序変更はローカル状態のみを変更し，保存操作まで永続化されない。

**Why this priority**: 並び替えはコンテンツ構成に必要だが，追加・編集・保存が
先に動作していなければ意味をなさないため P2 とする。

**Independent Test**: 複数ブロックを持つページで上下移動を行い，
順序が正しく入れ替わることを確認する。

**Acceptance Scenarios**:

1. **Given** 3 つ以上のブロックが表示されている, **When** 中間のブロックの上移動ボタンをクリックする, **Then** そのブロックが 1 つ上に移動し，他のブロックの順序が調整される
2. **Given** 3 つ以上のブロックが表示されている, **When** 中間のブロックの下移動ボタンをクリックする, **Then** そのブロックが 1 つ下に移動する
3. **Given** 先頭のブロック, **When** 上移動ボタンを確認する, **Then** 上移動ボタンが無効化されている
4. **Given** 末尾のブロック, **When** 下移動ボタンを確認する, **Then** 下移動ボタンが無効化されている
5. **Given** ブロックを並び替えた直後, **When** 画面の状態を確認する, **Then** 未保存インジケータが表示されている

---

### User Story 5 - ブロックを削除する (Priority: P2)

ユーザーがブロックの削除ボタンをクリックすると，そのブロックがエディタ画面から
除去される。削除はローカル状態のみを変更し，保存操作まで永続化されない。
誤操作防止のため，削除前に確認は行わない（未保存のため，保存前なら戻るボタンで
リロードすれば復元可能）。

**Why this priority**: ブロックの追加・編集が先に動作している前提で，不要なブロックの
除去が可能になる。

**Independent Test**: ブロックを削除し，画面から消えること，
残りのブロックの順序が維持されることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数のブロックが表示されている, **When** あるブロックの削除ボタンをクリックする, **Then** そのブロックが画面から消え，残りのブロックの順序が詰められる
2. **Given** ブロックが 1 件のみ, **When** そのブロックを削除する, **Then** ブロックが消え，空状態の案内が表示される
3. **Given** ブロックを削除した直後, **When** 画面の状態を確認する, **Then** 未保存インジケータが表示されている

---

### User Story 6 - 編集内容を明示的に保存する (Priority: P1)

ユーザーが保存ボタンをクリックするか Ctrl+S を押すと，エディタ画面上の全ブロック
（内容と順序）が一括で永続化される。保存が完了すると未保存インジケータが消え，
成功通知が表示される。保存に失敗した場合，エラー通知が表示され，
編集内容はローカル状態に保持される（データロスなし）。

**Why this priority**: 永続化なしではアプリを閉じた時点で全編集内容が失われるため，
追加・編集と並んで不可欠な機能。

**Independent Test**: ブロックを追加・編集・並び替え後に保存し，アプリを再起動
（ページを再度開く）して内容と順序が維持されていることを確認する。

**Acceptance Scenarios**:

1. **Given** 未保存の変更がある, **When** ユーザーが保存ボタンをクリックする, **Then** 全ブロックが永続化され，未保存インジケータが消え，成功通知が表示される
2. **Given** 未保存の変更がある, **When** ユーザーが Ctrl+S を押す, **Then** 保存ボタンと同じ動作で保存される
3. **Given** 保存が完了した, **When** ページを閉じて再度開く, **Then** 保存時の内容と順序が完全に再現される
4. **Given** 未保存の変更がある, **When** 保存処理が失敗する, **Then** エラー通知が表示され，編集内容はローカル状態に保持される
5. **Given** 変更がない状態, **When** 保存操作を行う, **Then** 不要な書き込みは行われず，保存済みである旨が表示される

---

### User Story 7 - 未保存状態を視覚的に確認する (Priority: P2)

ユーザーがブロックの追加・編集・並び替え・削除を行うと，画面上に未保存状態を示す
インジケータが表示される。保存が完了するとインジケータが消える。
未保存状態でページ一覧に戻ろうとした場合，確認ダイアログが表示される。

**Why this priority**: 未保存状態の可視化はデータ保護に直結するが，保存機能自体が
先に動作していなければ成立しないため P2 とする。

**Independent Test**: ブロックを変更して未保存インジケータが表示されること，
保存後に消えること，未保存状態での画面離脱時に確認が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ページを開いた直後（変更なし）, **When** 画面を確認する, **Then** 未保存インジケータは表示されていない
2. **Given** ブロックに何らかの変更を加えた, **When** 画面を確認する, **Then** 未保存インジケータが表示されている
3. **Given** 未保存の変更がある, **When** 保存を実行する, **Then** 未保存インジケータが消える
4. **Given** 未保存の変更がある, **When** 戻るボタンをクリックする, **Then** 「未保存の変更があります。破棄しますか？」という確認ダイアログが表示される
5. **Given** 確認ダイアログが表示されている, **When** キャンセルを選択する, **Then** エディタ画面に留まり，変更は保持される
6. **Given** 確認ダイアログが表示されている, **When** 破棄を選択する, **Then** ページ一覧に戻り，未保存の変更は破棄される

### Edge Cases

- **保存中のクラッシュ**: 一括保存はトランザクション内で実行されるため，クラッシュ時は
  前回の保存状態にロールバックされる。ユーザーのローカル編集状態は失われるが，
  永続化済みデータは破損しない。
- **大量ブロック**: 1 ページに 1,000 ブロックが存在する場合でも，一覧表示と保存操作が
  1 秒以内に完了すること。初期スコープでは仮想スクロールは導入しないが，
  パフォーマンスが許容範囲を超えた場合に後続対応として検討する。
- **ページ削除との整合性**: ページが削除された場合，そのページに属するブロックは
  CASCADE 削除により自動的に除去される。エディタ画面を開いている間にページが
  削除されるケースはローカルアプリのため発生しない（シングルユーザー）。
- **ブロック内容の上限**: 1 ブロックあたり 10,000 文字を上限とする。上限を超える入力は
  フロントエンドで制限し，ユーザーに通知する。
- **空ブロックの扱い**: 内容が空のブロックも有効なブロックとして保存される。
  ユーザーが意図的に空行を作ることを許容する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーがページ一覧でタイトルをクリックすると，エディタ画面に切り替わり，
  そのページのブロックが順序どおりに表示されなければならない（MUST）
- **FR-002**: エディタ画面に戻るボタンを設け，クリックでページ一覧画面に復帰できなければ
  ならない（MUST）
- **FR-003**: ユーザーがエディタ画面でプレーンテキストブロックを末尾に追加できなければ
  ならない（MUST）
- **FR-004**: ユーザーがブロックのテキスト内容を編集でき，変更が画面上に即座に反映
  されなければならない（MUST）
- **FR-005**: ユーザーが上移動・下移動ボタンでブロックの順序を変更できなければならない
  （MUST）
- **FR-006**: ユーザーがブロックを削除でき，残りのブロックの順序が自動で詰められなければ
  ならない（MUST）
- **FR-007**: 保存ボタンまたは Ctrl+S により，全ブロック（内容と順序）が一括で永続化
  されなければならない（MUST）。保存はトランザクション内で実行すること。
  バックエンドはブロック追加時点で UUIDv7 を生成し，EditorSession のインメモリ状態を
  トランザクションで永続化すること（MUST）。
- **FR-008**: 未保存の変更がある場合，未保存インジケータを画面上に表示しなければならない
  （MUST）
- **FR-009**: 未保存の状態でページ一覧に戻ろうとした場合，確認ダイアログを表示しなければ
  ならない（MUST）
- **FR-010**: すべてのブロック操作はローカルで完結し，外部通信を一切行ってはならない
  （MUST NOT）
- **FR-011**: ブロックの内容は空文字を許容し，最大 10,000 文字に制限しなければならない
  （MUST）
- **FR-012**: ブロックの順序は整数値（position）で管理し，保存時にフロントエンドの
  配列順序を 0 始まりで振り直さなければならない（MUST）
- **FR-013**: ブロック追加時に新しいブロックに自動でフォーカスを当てなければならない
  （MUST）
- **FR-014**: 保存失敗時にエラー通知を表示し，ローカルの編集状態を保持しなければ
  ならない（MUST）。保存失敗によりユーザーの編集内容が失われてはならない（MUST NOT）。

### Key Entities

- **Block**: ページに属するテキストコンテンツの最小単位。UUIDv7（バックエンドで生成）で
  識別され，所属ページ（page_id），種別（block_type, デフォルト 'text'），
  内容（content），表示順序（position），作成日時，更新日時を持つ。
  ページが削除されるとブロックも連鎖的に削除される。
  空の内容を許容し，1 ブロックあたり最大 10,000 文字。

## Constraints & Compliance *(mandatory)*

- **CC-001 Data Integrity**: ブロックの一括保存はトランザクション内で実行する。
  途中で失敗した場合は全変更がロールバックされ，前回の保存状態が維持される。
  マイグレーションにより blocks テーブルを追加し，既存の pages テーブルとの
  外部キー制約（ON DELETE CASCADE）でデータ整合性を保証する。
- **CC-002 Privacy**: ブロックの追加・編集・保存・削除はすべてローカルの SQLite
  データベース内で完結する。外部通信，テレメトリ，サーバー依存機能を導入しない。
- **CC-003 Performance**: 1,000 ブロックを持つページの表示が 1 秒以内に完了すること。
  1,000 ブロックの一括保存が 1 秒以内に完了すること。個別のブロック操作
  （追加・編集・並び替え・削除）は画面上で遅延なく反映されること。
- **CC-004 Boundary Types**: ブロックデータは型付き IPC 契約を通じて Rust と
  TypeScript の間で受け渡す。ブロックの内容検証はドメイン層で行い，
  IPC 層ではエラーをシリアライズ可能な形式で返す。
- **CC-005 Testability**: ブロックのドメインロジック（値オブジェクト検証），
  リポジトリ操作（CRUD + 一括保存），IPC コマンドの各層で独立にテスト可能であること。
  In-memory SQLite を用いた統合テストで永続化の正確性を検証する。

## Assumptions

- ブロック種別は現時点では「テキストブロック」のみ。将来的に見出し・リスト・画像等の
  ブロック種別を追加する可能性があるが，本スコープでは対象外とする。
  初期スキーマに `block_type TEXT NOT NULL DEFAULT 'text'` カラムを含め，
  将来の種別追加時のマイグレーションを不要にする。
- 自動保存（デバウンス）は本スコープでは対象外。将来の拡張として検討する。
- ドラッグ＆ドロップによる並び替えは本スコープでは対象外。上下移動ボタンで代替する。
- フロントエンドのルーティングライブラリは導入しない。React の状態管理による
  画面切り替えで実装する。
- ブロック削除時の確認ダイアログは設けない。未保存のため，戻るボタンで
  リロードすれば復元可能であることを根拠とする。
- ページのタイトル編集機能は 001-page-persistence で実装済みであり，
  エディタ画面からもタイトルを編集可能にするかは本スコープでは対象外とする。

## Scope Boundaries

### 対象範囲内

- ページ一覧 ↔ エディタの画面切り替え
- プレーンテキストブロックの追加・内容編集・並び替え（上下移動）・削除
- 明示的保存（ボタン + Ctrl+S）による一括永続化
- 未保存状態のインジケータ表示
- 未保存時の画面離脱確認

### 対象範囲外

- リッチテキスト（見出し，リスト，太字，コード等）
- ブロック種別の選択・変換
- 自動保存（デバウンス方式）
- ドラッグ＆ドロップによる並び替え
- クライアントサイドルーティング（React Router 等）
- 仮想スクロール
- Undo / Redo
- エディタ画面からのページタイトル編集
- キーボードショートカット（Ctrl+S 以外）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがページを開き，ブロックを追加・編集し，保存して再度開くまでの
  一連の操作が 30 秒以内に完了できること
- **SC-002**: 1,000 ブロックを持つページの表示および一括保存がそれぞれ 1 秒以内に
  完了すること
- **SC-003**: 保存失敗時にユーザーの編集内容が保持され，エラー内容が通知され，
  再試行が可能であること
- **SC-004**: すべての品質ゲート（`cargo make qa`）を通過し，各ユーザーストーリーの
  受入シナリオが検証可能であること
