# 概要

このプロジェクトでは，Notonにインスパイアされたローカルノートブックアプリを開発する。

# 背景

- 環境によっては自由にSaaSサービスが使えず，よいメモアプリが使用できない。
- リスト・カンバン・ガントチャートとビューを切り替えてタスク管理できるツールがない

# 機能要件

以下のような特徴を有数するデスクトップアプリを開発する。

## 1. ブロック指向アーキテクチャ (Block-based Architecture)

ドキュメントは単一のテキストデータ（巨大なString）ではなく、独立した「ブロック」の連なりとして表現されます。

- すべてがブロック: 段落、見出し(H1~H3)、To-Doリスト、画像、データベースなど、画面上のあらゆる要素が「ブロック（Node）」という最小単位です。
- データ構造: 各ブロックは一意のID（UUID等）、タイプ（text, heading, todoなど）、コンテンツ（テキストやメタデータ）、そしてツリー構造を形成するための親ID・子IDのポインタを持ちます。

利点: これにより、後述するドラッグ＆ドロップでの自由な配置換えや、ブロック単位でのプロパティ変更が容易になります。

## 2. 「すべてがページ」と無限階層 (Page & Tree Structure)

ファイルシステムのディレクトリ（フォルダ）と、ドキュメント（ファイル）の境界を無くした特徴です。

- ページもブロックの一種: 「ページ」という概念自体も、特別なメタデータ（タイトル、アイコン絵文字、カバー画像）を持った最上位のブロックとして扱われます。
- 無限のネスト: ページの中にページ（Sub-page）を無制限に作成でき、ツリー状の階層構造を構築できます。
- インラインとフルページ: ページは、他のテキストの中に埋め込まれる（リンクとして機能する）ことも、独立した画面として開くこともできます。

## 3. 多面的なデータベースとビュー (Database & Views)

「ボード、リスト、ガントチャート」を実現するための最重要機能です。Notionのデータベースは、単なる表ではなく、リレーショナルデータベースの概念をUIに落とし込んだものです。

- レコード ＝ ページ: データベースの1行（レコード）は、それ自体が開くことのできる「ページ」です。これにより、タスクの「タイトル」や「ステータス」を一覧で管理しつつ、詳細なメモをレコードの中（ページ内）に書くことができます。
- プロパティ (カラム): 各データベースは、テキスト、数値、選択肢（Select）、日付（Date）、チェックボックスなどの型付けされたプロパティ（スキーマ）を定義できます。
- ビュー (Views): 同じデータソースに対して、異なるフィルタ・ソート・レンダリングを適用して表示を切り替える機能です。
- リスト (List): プロパティを最小限にして、ドキュメントの目次のようにシンプルに表示。
- ボード (Board / Kanban): 特定の「選択肢」プロパティ（例：ステータス）を軸にして列（Column）を作り、カードを分類。
- ガントチャート (Timeline): 「日付」プロパティ（開始・終了）を参照し、時間軸上にタスクの長さをバーとして描画。

## 4. シームレスなUI/UX (Rich Interactions)

ユーザーが直感的に、かつキーボードから手を離さずに操作できる工夫です。

- スラッシュコマンド (/): エディタ上で / を入力するとポップアップが開き、ブロックのタイプ変換や新しい要素の挿入が即座に行えます。
- ドラッグ＆ドロップ (D&D): すべてのブロック（段落やタスク、データベースの行まで）は、左側のハンドル（⋮⋮ のようなアイコン）を掴んで自由に順序を入れ替えたり、他の列に移動させたりできます。
- マークダウンの自動変換: *  で箇条書き、#  で見出しなど、入力した瞬間に対応するブロックタイプに変換されます。

# 非機能要件

## 1. パフォーマンスと応答性 (Performance)

TauriとRustを採用する最大のメリットを活かす領域です。本家Notionで時折ストレスになる「読み込み待ち」をゼロにすることが目標になります。

- 起動時間: アプリケーションの起動から初期画面（ダッシュボードや直前のページ）の描画完了までを 1〜2秒以内 とする。
- UIの応答性: ブロックの追加、タイピング、ドラッグ＆ドロップなどの操作において遅延を感じさせない（理想は 60fps / 16ms以内の描画完了）。
- レンダリング最適化: 大規模なデータベース（例：数千件のタスク）のリストビューやガントチャートを開く際、DOMの肥大化を防ぐために 仮想スクロール（Virtualization） を必須要件とする。
- メモリフットプリント: 常駐アプリとしてバックグラウンドで起動しておけるよう、アイドル時のメモリ使用量を最小限に抑える（WebViewのプロセスを含めても軽量に保つ）。

## 2. データ完全性と保護 (Data Integrity & Backup)

クラウド同期がない完全ローカルアプリにおいて、「ユーザーのデータを絶対に消失させないこと」 が最重要の非機能要件となります。

- ACID特性の担保: データベース（SQLite）への書き込みはRust側で厳格にトランザクション管理を行い、アプリのクラッシュ時でもデータが破損しないようにする。
- 自動バックアップ機構: ユーザーが意識しなくても、定期的に（例：アプリ起動時や1日に1回）SQLiteのデータベースファイルのコピーを別のローカルディレクトリ（例：Documents/AppBackups）に自動退避する機能を実装する。
- ファイルシステムのエラーハンドリング: ディスク容量不足や、添付ファイル保存時のパーミッションエラーなどを適切に捕捉し、ユーザーに分かりやすい形で警告を出す。

## 3. 保守性と拡張性 (Maintainability)

将来的な機能追加や、UIフレームワークの変更に耐えうる設計が必要です。

- フロントとバックの疎結合: フロントエンド（React/Vue）とバックエンド（Rust）間の通信（Tauri IPCコマンド）は、APIのインターフェースとして明確に型定義（TypeScriptの型とRustの構造体を同期）する。
- データベースのマイグレーション: アプリのバージョンアップに伴いデータベースのスキーマが変更される場合に備え、Rust側で起動時に自動でスキーマを更新するマイグレーションツール（例：sqlx-cliの機能など）を組み込む。

## 4. セキュリティとプライバシー (Security)

ローカル完結であること自体が強力なセキュリティ要件を満たしています。

- 外部通信の遮断: アプリケーション本体から外部サーバーへのテレメトリ（利用状況の送信）や意図しない通信を行わない。
- データ保存の分離: OSのユーザープロファイル領域（Windowsの AppData\Local や AppData\Roaming）にデータベースと設定ファイルを保存し、OSの標準的なユーザー権限によって他のユーザーアカウントからデータが保護されるようにする。

## 5. 配布と実行環境 (Deployment)

Windowsユーザーが簡単に導入・使用できるようにします。

- 対応OS: Windows 10 および Windows 11 (64-bit) のみをターゲットとする。
- インストーラー形式: Tauriの標準バンドラーを使用し、依存関係（WebView2ランタイムなど）を自動で解決できる単一のインストーラー (.msi または .exe 形式) を提供する。
- スタンドアロン動作: 管理者権限（UAC）がなくてもインストール・実行が可能な構成にする（AppDataへのインストールなど）。

# 技術スタック

- バックエンドはRust（Tauri）とする。
- フロントエンドはTypescript（React/Vue）とする。
  - Typescriptのパッケージ管理にはpnpmを使用する。

# 実装規約

## Rustの安全性

- バックエンドのコードベースはRust（エディション2024）で記述する必要がある。すべてのコードは業務利用可能な品質基準を満たす必要がある。
- このコードベースでは、例外なく「unsafe」コードブロックを使用してはなりません。
  - 依存関係がFFI境界で「unsafe」を必要とする場合、その相互作用は依存関係クレート自体にカプセル化されなければならず、アプリケーションコード内ではカプセル化されません。
- このプロジェクトはRust学習の演習も兼ねています。コードは、巧妙な抽象化よりも、明快さと慣用的なパターンを優先する必要があります。

**理由**: Rustの所有権モデルは、業務データを扱うツールにとって極めて重要なメモリ安全性を保証します。絶対的な「unsafe」禁止により、潜在的な欠陥のクラス全体が排除され、コードレビューが簡素化されます。学習目標は、開発者の成長を支援するために可読性を優先することを意味します。

## ドメイン駆動設計

すべての設計成果物（仕様、計画、モデル）は、ドメイン駆動設計（DDD）の原則に従う必要があります。

- **ユビキタス言語**: ドメイン用語は、コード、ドキュメント、UI全体で一貫して使用する必要があります。開発者は、ドメイン語彙から逸脱する同義語や略語を作成してはいけません。
- **境界付きコンテキスト**: Cargoワークスペースの各クレートは、境界付きコンテキストを表します。適切な抽象化によってクレートを分割してください。
- **エンティティと値オブジェクト**: ドメインオブジェクトは、エンティティまたは値オブジェクト (不変、値による等価性) としてモデル化する必要があります。
- **集約**: 集約ルートを適切に設定してください。

**理由**: 業務フローには明確に定義されたドメイン語彙があります。DDD は、コードベースがドメイン専門家の言語を反映していることを保証するため、誤解を減らし、開発者以外の利害関係者にもシステムを理解できるようにします。

## テスト駆動開発

実装は必ず Red-Green-Refactor サイクルに従う必要があります。

1. **赤**: 実装コードを書く前に、期待される動作を定義する失敗するテストを記述します。
2. **緑**: テストに合格するための最小限のコードを記述します。
3. **リファクタリング**: すべてのテストをグリーンのままにしてクリーンアップします。

コードをコミットする前に、タスクランナーのパイプライン全体を通過させる必要があります。パイプラインは `Makefile.toml` で定義され、`cargo make watch` または個別に実行されます。

- `cargo make fmt` — すべてのコードをフォーマットします (`cargo fmt --all`)
- `cargo make clippy` — すべてのターゲットを lint します (`cargo clippy`)
- `cargo make test` — すべてのテストを実行します (`cargo nextest run`)
- `cargo make qa` — `fmt`、`clippy`、`test` を順に実行します

これら3つのステップのいずれかが失敗した場合、コミットを作成してはいけません。これにより、すべてのコミットにおいて、フォーマットの一貫性、リントフリーなコード、そして回帰のない実装が保証されます。

**根拠**: TDDは、構築段階からテスト可能な設計を生み出し、回帰を防ぎます。必須のfmt→clippy→testゲートにより、すべてのコミットが一貫した品質基準を満たすことが保証されます。

## ソフトウェア工学の基礎

すべてのコードは、次の基本原則に従う必要があります。

- **固体**：
  - *単一責任*: 各モジュール/構造体には、変更する理由が 1 つなければなりません。
  - *オープンクローズ*: 既存のコードを変更するのではなく、特性を通じて動作を拡張します。
  - *Liskov Substitution*: 特性の実装は、呼び出し元を壊すことなく置換可能でなければなりません。
  - *インターフェースの分離*: 特性は小さく、焦点を絞ったものでなければなりません。クライアントは使用しないメソッドに依存してはなりません。
  - *依存性の逆転*: 高レベルモジュールは、具体的な実装ではなく、抽象化 (特性) に依存する必要があります。
- **KISS**: 要件を満たす最もシンプルなソリューションを優先します。早まった抽象化、不要なジェネリック化、過剰なエンジニアリングは避けます。
- **YAGNI**: 実際に必要になるまで機能を実装しないでください。投機的な機能は禁止されています。
- **DRY**: 3箇所以上で重複が存在する場合にのみ、共有ロジックを抽出します。早期の抽象化を避けるため、重複が2箇所までであれば許容されます。

**根拠**: これらの原則は、保守性と理解性に優れたコードを生み出します。学習プロジェクトにおいては、基本を習得することは機能を提供することと同じくらい重要です。

## Rustドキュメンテーション

すべてのパブリックアイテム（`pub fn`、`pub struct`、`pub enum`、`pub trait`、`pub type`、`pub mod`）には、`///` ドキュメントコメントが必要です。クレートルート（`lib.rs`）には、`//!` モジュールレベルのドキュメントが必要です。プライベートアイテムは、名前とシグネチャから目的がすぐに分からない場合は、ドキュメントコメントを付けるべきです。

すべてのドキュメントコメントは次の構造に従う必要があります:

1. **概要行** (必須): アイテムの機能を説明する簡潔な一文。この行はモジュールの概要と検索結果に表示されるため、自己完結的でなければなりません。
2. **詳細な説明**（必要な場合）: 追加のコンテキスト、エッジケース、不変条件、または使用上の注意。概要で十分な場合は省略してください。
3. **セクション**（該当する場合）:
   - `# Examples` は、すべての public 関数およびメソッドに必ず記述してください。サンプルコードは、doc-test としてコンパイルおよび実行する必要があります。定型コードには、`# fn main() -> Result<...> { ... }` のような隠し行を使用してください。`no_run` は、I/O 依存のサンプルコードにのみ使用してください。`ignore` は、明示的な理由がない限り使用しないでください。
     - **例外 - 単純なアクセサメソッド**: 単純なフィールドアクセサ (ロジック、分岐、副作用のない、単一フィールドの参照またはコピーを返すゲッター) であるパブリックメソッドは、`# Examples` の MUST 要件から免除されます。これらのメソッドには、引き続き `///` サマリー行が必要です。理由: コンストラクターが `pub(crate)` の場合、ドキュメントテスト (外部コードとしてコンパイルされる) は型を構築できないため、カプセル化を弱めることなくコンパイル可能な例を作成することはできません。単純なゲッターの定型例は、意味のある使用パターンを示すことなく、ノイズを追加します (原則 VIII KISS、XI 保守性第一)。この免除は、フィールドアクセスを超えるロジックを含むメソッド、計算または変換された値を返すメソッド、パブリックコンストラクターまたはビルダーメソッド、またはロジックから派生した `Result` または `Option` を返す関数には適用されません。これらの免除されないメソッドには、引き続きコンパイル可能な `# Examples` が必要です。
   - `# Errors` — `Result` を返す関数には必ず記述してください。各エラーの種類と、その発生条件を文書化してください。
4. **ドキュメント内リンク**: 関連する型、関数、モジュールを相互参照するには、`[`TypeName`]` 構文を使用します。散文的な参照よりもドキュメント内リンクを優先してください。

追加ルール:

- アイテムのドキュメントには、ブロック スタイル (`/** */`) ではなく、線スタイル (`///`) を使用します。
- コンパイル時にドキュメント化されていないパブリック項目をキャッチするために、すべてのクレート ルートで `#![warn(missing_docs)]` を有効にします。
- 品質ゲートの一部として `cargo doc --no-deps` を実行し、警告なしでドキュメントがビルドされることを確認します。
- ドキュメントコメントは英語で記述する必要があります (すべての成果物を英語で記述するというルールに準拠)。

**理由**: 豊富なドキュメントはコードベースを自己記述的にし、オンボーディングを加速し、`cargo test --doc` で壊れた例を検出します。学習プロジェクトでは、ドキュメントを作成することで各APIの契約と意図の理解が深まります。

## 防御的なエラー処理

アプリケーションはいかなる状況下でも実行時にパニックを起こしてはなりません（MUST NOT）。誤りが発生する可能性のあるすべての操作は `Result<T, E>` を返し、呼び出し元にエラーを伝播しなければなりません（MUST）。以下の構文はアプリケーションコードでは**禁止**されています。

| 禁止 | 代替手段が必要 |
|---|---|
| `.unwrap()` | `.map_err()?`、`match`、または `.unwrap_or_default()` |
| `.expect("...")` | `.map_err()?` と `thiserror` による型付きエラー |
| `panic!()` | `Err(...)` を返します |
| `unreachable!()` | `Err(...)` または完全一致を返します |
| `todo!()` | コンパイル時の `unimplemented!()` は、`dev` にマージしてはならないプロトタイピング ブランチでのみ許可されます |
| テスト以外のコードで `assert!()` を実行する | 検証エラーで `Err(...)` を返す |

**許可される例外** (テスト コードのみ):

- `#[cfg(test)]` モジュール内および `tests/` 下の統合テストファイル内では、`unwrap()`、`expect()`、`assert!()`、`panic!()` の使用が許可されます。テストが失敗した場合は、パニックが発生することが想定されます。

**エラー設計ルール**:

- すべてのドメイン エラーは、`api` クレート内の `thiserror` 列挙型として定義する必要があります (原則 VI - 境界付けられたコンテキストを参照)。
- エラーバリアントには、意味のあるユーザー向けメッセージを生成するのに十分なコンテキストが含まれている必要があります (例: ファイル パス、フィールド名、予想される値と実際の値)。
- エラー伝播には `?` 演算子を使用する必要があります。`Result` に対する手動の `match` は、変換または回復が必要な場合にのみ許可されます。
- 関数はエラーを黙って無視してはいけません。エラーを意図的に無視する場合は、コードに「// INTENTIONAL: <reason>」というコメントを必ず付けてください。

**クリッピーの施行**:

違反を自動的にキャッチするには、次の Clippy リントをワークスペース レベルで有効にする必要があります。

```toml
# Cargo.toml (workspace)
[workspace.lints.clippy]
unwrap_used = "deny"
expect_used = "deny"
panic = "deny"
unreachable = "deny"
```

```toml
# Each crate's Cargo.toml
[lints]
workspace = true
```

**理由**: 業務データの整合性には、予測可能で回復可能なエラー動作が求められます。本番環境でパニックが発生すると、進行中のデータ操作が破損したり、ユーザーが対処できないままアプリケーションがクラッシュしたりする可能性があります。`unwrap` と `panic` を禁止することで、エラーが発生する可能性のあるすべての呼び出しサイトで明示的なエラー処理が強制され、障害モードが可視化され、テストされ、回復可能なソフトウェアが実現します。

## 保守性を第一に

読みやすさ／保守性と実行効率が相反する場合は、読みやすさ／保守性を優先する必要があります。アプリケーションは、高スループットのサーバーではなく、ローカルまたはクラウド同期フォルダーで少数のExcelファイルを管理しているため、マイクロ最適化はユーザーにとって目に見えるメリットがほとんどなく、認知負荷と不具合のリスクが増大します。

具体的なルール:

- プロファイリングの証拠によって測定可能なユーザー向け問題が示されない限り、パフォーマンスの最適化 (キャッシュ、遅延評価、カスタム アロケータ、ビット パッキングなど) を導入してはなりません。
- 読みやすさを低下させる巧妙な分岐、ネストされたクロージャ、または高度な型レベルのトリックよりも、単純で線形の制御フローを優先する必要があります。
- クローンされたデータが小さく、ライフタイムの複雑さによって意図が不明瞭になる場合は、複雑なライフタイム アノテーションよりも `clone()` を優先する必要があります。
- チェーンが 3 つのコンビネータを超える場合、または明らかでないクロージャが必要な場合は、長い反復子チェーンよりも、名前付き変数を使用した明示的なループを優先する必要があります。
- より単純な所有値の代替が文書化された正当な理由によって除外されていない限り、`Cow`、`Rc`、`Arc`、または内部可変性パターン (`RefCell`、`Cell`) を使用しないでください。
- 2 つの実装が機能的に同等である場合、Rust 初心者が初めて読んで理解できる実装を選択する必要があります。

**理由**: これは、Rustを学習中の個人開発者が管理する小規模なデスクトップツールです。このようなプロジェクトでは、時期尚早な最適化が予期せぬ複雑さの主な原因となります。コードを簡潔で扱いやすい状態にしておくことで、将来的な変更（同じ開発者が数ヶ月後に行う場合も、新しいコントリビューターが行う場合も）を自信を持って迅速に行うことができます。アプリケーションのワークロード（数十個のファイル、1秒未満の処理）は、複雑なRustイディオムによって実現されるパフォーマンスの限界を超えることはありません。
